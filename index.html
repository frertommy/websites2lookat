Critical Gaps — Must Fix Before Any User Touches This

BLOCKERS

Hardcoded Helius API Key Leaked in 6+ Config Files

The key 3f2a657d-6196-4596-afa2-ee05787d778b is hardcoded in plain text across custom-dex.config.yaml, custom-dex-filler.config.yaml, sportsmaker-tight.config.yaml, and the chain-monitor defaults. This isn't just a devnet convenience — if this repo goes public or the key has mainnet-beta RPC access, it's immediately exploitable. All RPC endpoints need to be env vars, zero hardcoded keys.

1 HOUR

No Privy / Embedded Wallet — Still on Raw wallet-adapter

Frontend uses @solana/wallet-adapter-react directly. Zero Privy integration. The entire Notch.win UX thesis (Hyperliquid-like mobile, gasless orders, one-tap onboarding) depends on embedded wallets. This is the biggest UX gap — a retail user hitting "Connect Wallet" and needing Phantom is a 90%+ drop-off. Privy SDK needs to wrap the wallet provider, handle account creation, and bridge to the Drift SDK's Wallet interface.

2-3 WEEKS

No Gasless / Swift Order Path from Frontend

The Swift server is in the repo (Rust service with Kafka), and the keeper has placerExample.ts for Swift order placement — but the frontend has zero Swift integration. Every trade currently goes through a normal Solana transaction that the user signs and pays gas for. For the "mobile-first gasless" experience, the frontend needs to: sign an off-chain Swift message → POST to the Swift server → server submits on-chain. This is the other half of the Privy story.

2-3 WEEKS

No Mainnet Program Keypair — No Deployment Pathway

The keys/ directory is empty (just .gitkeep). There are no mainnet keypairs, no deploy scripts for mainnet-beta, and Anchor.toml still points at a localnet program ID (6prdU12b...) that differs from the devnet ID (2XUSofeG...). The deployment.json only defines dev and staging — no production environment. Need: mainnet program keypair generation, Anchor.toml alignment, and a deployment runbook.

1 WEEK

Fee Structure Not Customized — Still Drift's 5bps Defaults

The on-chain program hardcodes Drift's original fee tiers (5/4.5/4/3.5/3/2.5 bps taker, 2bps maker rebate, 6 volume tiers). The Notch target is 10bps taker / 2bps maker with no rebates. The update_perp_fee_structure admin instruction exists but there's no script to call it. Need a one-shot admin script to set the production fee schedule post-deploy, and decide whether to keep volume tiers or flatten to a single tier.

1 DAY

Anchor.toml Program ID Mismatch

Anchor.toml declares 6prdU12b... for all networks, but lib.rs has declare_id!("2XUSofeG...") and all services reference 2XUSofeG.... This mismatch means anchor build and anchor deploy won't work correctly without manual overrides. These need to be aligned — either update Anchor.toml to match the deployed devnet program, or clarify which is canonical.

Not Started — Needs Building

MISSING
● NOT STARTED

Account Struct Optimization (Min SOL Rent)

Drift's User account is ~4,000+ bytes. At mainnet rent rates that's ~0.03 SOL per account. The plan was to optimize struct packing to minimize rent for new users (critical for embedded wallet onboarding where Notch pays the rent). No work done — the account struct is unchanged from upstream Drift. This requires protocol-level Rust changes.

● NOT STARTED

Mobile App / PWA

The entire Notch.win thesis is mobile-first trading. The current frontend is a Next.js web app with no PWA manifest, no service worker, no mobile-specific layouts, no touch gesture handling for the trading chart. Building a Swift wrapper (or React Native) for iOS with Privy embedded wallets is a separate workstream that hasn't started.

● NOT STARTED

USDC Onramp Integration

Retail users need to buy USDC to trade. There's no fiat onramp integration (MoonPay, Transak, Stripe crypto). The devnet has a faucet script but production users need a real path from card/bank → USDC → deposit. This is a major UX cliff.

● NOT STARTED

Offseason / Season Transition Logic

What happens when EPL season ends? The oracle pipeline currently has no concept of offseason pricing, league transitions, promotion/relegation, or market settlement/expiry. The V2 oracle design included offseason futures regime using Betfair bracket probabilities — none of that is implemented in the team-oracle service. Markets will just go stale.

● NOT STARTED

Multi-League Expansion

Config currently has 14 teams (10 EPL + 3 La Liga + 1 EPL extra). The MSI API serves 96 teams across multiple leagues. No UI for league switching, no config for dynamically adding markets, no admin scripts for initializing new perp markets post-deploy. Adding Serie A or Bundesliga is manual Rust/admin work today.

● NOT STARTED

Security Audit

Drift's upstream was audited by Neodyme. Your fork inherits that audit for unchanged code, but any modifications (new oracle types, custom fee logic, removed features) invalidate coverage. The PrelaunchOracle admin-key dependency is itself a centralization risk that an auditor would flag. No audit planned or scoped.

Recommended Execution Order

ROADMAP

The two critical paths are: (A) mainnet deployment readiness, and (B) retail UX. They can run in parallel.

Path A is infrastructure — keys, fees, configs, security. Path B is Privy + Swift + mobile. Path A unblocks testnet trading with real users. Path B is what makes Notch different from "yet another perp DEX with Phantom wallet."
